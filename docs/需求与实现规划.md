## 1. 项目背景

本项目面向青少年 C++ 编程能力培养。目标不是让学生长期停留在枯燥的刷题模式，而是通过更形象的算法演示、更生动的讲解、更有效的练习，以及基于学生反馈的持续迭代，让抽象算法变得直觉可感。

核心理念：

```text
看见算法运行过程 -> 理解代码结构 -> 动手练习 -> 获得反馈 -> 修正认知 -> 再次练习
```

## 2. 项目目标

### 2.1 产品目标

1. 让学生能通过视频和交互内容理解算法的状态变化。
2. 让学生能把动画中的对象对应到 C++ 代码中的变量、数组、循环和递归调用。
3. 让学生通过临时 OJ 完成练习、获得即时反馈。
4. 让教师或内容制作者能持续扩展章节、视频、讲义和题目。
5. 为后续接入某公司的在线 OJ 预留接口，避免平台重写。

### 2.2 教学目标

1. 降低算法入门的抽象门槛。
2. 培养学生的算法直觉，而不只是记忆模板。
3. 帮助学生从“我会背代码”过渡到“我知道它为什么这样运行”。
4. 用反馈数据发现学生卡点，反向优化讲解和题目设计。

## 3. 用户角色

### 3.1 学生

学生需要完成：

- 浏览章节目录。
- 观看 Manim 算法演示视频。
- 阅读算法讲解和 C++ 代码拆解。
- 在线编写并提交 C++ 代码。
- 查看判题结果、样例反馈、错误提示。
- 回看错题和历史提交。

### 3.2 教师 / 内容维护者

教师或内容维护者需要完成：

- 新增和编辑章节、课时、讲义。
- 管理 Manim 视频和预览图。
- 管理题目、样例、隐藏测试点和难度。
- 查看学生学习数据和题目通过率。
- 根据反馈调整讲解和练习。

### 3.3 管理员

管理员需要完成：

- 管理用户、课程、题库和系统配置。
- 管理临时 OJ 的安全参数。
- 后续配置外部 OJ 接入。

## 4. 内容结构

每个算法知识点统一包含 5 个部分：

1. 情境引入：这个算法解决什么问题。
2. 动画演示：使用 Manim 展示算法运行过程。
3. 代码拆解：用 C++ 代码逐段解释变量、循环、判断和状态变化。
4. 练习题：学生通过 OJ 提交代码。
5. 反馈迭代：基于通过率、错误类型和学生反馈优化内容。

推荐的课时结构：

```text
章节 Chapter
  -> 课时 Lesson
    -> 概念 Concept
    -> 视频 Video
    -> 图文讲义 Notes
    -> 示例代码 Code Walkthrough
    -> 练习 Problems
    -> 学习反馈 Feedback
```

## 5. 九大章节细化

### 第一章：高精度计算

核心知识点：

- 为什么普通整数类型不够用。
- 用字符串或数组存储大整数。
- 数字反向存储的原因。
- 高精度加法。
- 高精度减法。
- 大数比较与符号处理。
- 高精度乘低精度。
- 高精度乘高精度。
- 高精度除低精度。
- 进位、借位、前导零处理。

适合 Manim 演示的内容：

- 普通整型容量边界和大数溢出。
- 字符串字符如何转换成可计算的数字数组。
- 反向存储如何让个位对齐下标 0。
- 竖式加法中的逐位相加和进位。
- 竖式减法中的逐位借位。
- 乘法中每一位贡献如何累加到结果数组。
- 前导零如何被清除。

练习方向：

- 判断整型范围与溢出。
- 字符串拆位、各位数字和、数字频次统计。
- 反向存储和低位下标访问。
- 大整数加法。
- 大整数减法。
- 阶乘结果输出。
- 高精度斐波那契。
- 高精度乘法。

- 冒泡排序。
- 选择排序。
- 插入排序。
- 计数排序。
- 快速排序初步。
- 归并排序初步。
- 稳定性。
- 时间复杂度和空间复杂度。

适合 Manim 演示的内容：

练习方向：

- 成绩排序。
- 结构体排序。
- 第 k 小数。
- 去重排序。
- 逆序对入门。

### 6.4 临时 OJ 系统

MVP 阶段需要一个可用但边界清楚的临时 OJ。

基本流程：

1. 前端提交题目 ID 和 C++ 源码。
2. 后端创建一次性提交记录。
3. 后端编译代码。
4. 后端按测试点运行程序。
5. 对比标准输出。
6. 返回判题结果。

判题状态：

- Pending。
- Judging。
- Accepted。
- Wrong Answer。
- Compile Error。
- Runtime Error。
- Time Limit Exceeded。
- Memory Limit Exceeded。

MVP 安全要求：

- 编译和运行必须隔离。
- 设置运行时间限制。
- 设置输出长度限制。
- 禁止无限制访问宿主文件系统。
- 为后续 Docker 沙箱预留接口。

建议设计 Judge Adapter：

```text
JudgeService
  -> LocalJudgeAdapter
  -> ExternalOjAdapter
```

这样后续接入公司在线 OJ 时，前端和课程内容不需要大改。

当前已落地：

```text
JudgeService
  -> LocalJudge
  -> DockerJudge
```

- `GET /api/judge` 可查询当前判题器能力。
- 当前默认判题器为 `local-process`，隔离级别为 `process-timeout-only`。
- 可通过 `JUDGE_ADAPTER=local-process` 或 `JUDGE_ADAPTER=docker` 选择适配器。
- 已新增 Docker 判题器骨架；本机 Docker daemon 当前未启动，仍需在可用环境中实测。
- 生产前仍需加固 Docker 沙箱或接入外部 OJ Adapter。

## 12. 视频设计规范

每个 Manim 视频建议遵循：

1. 开头 5 到 10 秒说明算法解决的问题。
2. 中间用数据结构动画展示状态变化。
3. 关键步骤出现文字提示。
4. 变量名要和代码一致。
5. 最后展示伪代码和复杂度。
6. 控制时长，单个知识点尽量 2 到 5 分钟。

视觉风格延续当前冒泡排序原型：

- 深色背景。
- 高亮比较区域。
- 橙色表示交换或变化。
- 绿色表示已完成或已确定状态。
- 蓝色、粉色用于指针或角色区分。

逐支视频的详细分镜、代码映射和练习衔接见 `docs/算法视频分镜规划.md`。
